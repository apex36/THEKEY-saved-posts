# Design decisions, trade-offs, and next steps

## 1. Key design decisions

**Single-row soft-delete `saved_posts` with `UNIQUE(user_id, post_id)`.** I modeled a bookmark as one row per `(user, post)` pair that lives forever, toggled by a nullable `deleted_at`. Saving sets `deleted_at = NULL` and bumps `saved_at`; un-saving sets `deleted_at = now()` and leaves the row in place. The requirement that re-saving must *reactivate* the existing record rather than create a new one falls directly out of this shape — there's physically nowhere else for a second save to go. "No duplicate active saves" is then a corollary, not a separately-enforced rule: with one row per pair, forever, there's no such thing as a duplicate row to begin with. The same full unique constraint is also my concurrency backstop (more below), which is why I used a full unique index rather than a partial one scoped to active rows — a partial index would let two soft-deleted-then-recreated rows exist for the same pair, and would only catch races on *active* saves, not the underlying invariant I actually want.

**Auth is an Elysia `resolve` that reads `x-user-id` and looks up the role from the `users` table — never a client-supplied role header.** The take-home stub explicitly allows a header/token/session stand-in for identity, but I didn't want the role itself to come from anywhere the client controls. If `role` were a header, any request could just claim `x-role: moderator` and bypass every 403 in the system, which defeats the point of "enforce access rules as if it were real." So the client only asserts *who* it is; the server is the sole source of truth for *what that user is allowed to do*. This is a `.resolve()` that runs before every route (no unauthenticated endpoint exists in this API, including the demo `/api/users` listing), which is what gives me the blanket 401 behavior.

**Pure decision functions (`decideSave` / `decideUnsave`) plus repository interfaces.** `domain/saved-posts.ts` has zero imports and zero I/O — it's a state machine over `existing: SaveSnapshot | null` that returns `'create' | 'reactivate' | 'noop'` (and the mirror for unsave). All of idempotency, reactivation, and count-behavior are unit-tested against this function directly, no database involved. The service layer (`ForumService`) is the only thing that calls it, using a `Repos` interface it doesn't know is backed by Drizzle in production and an in-memory fake in tests. That split is what let me write "double-save is a strict no-op" and "re-save reactivates the same row id" as fast, deterministic unit tests instead of DB-round-trip tests.

That said, a decision function operating on a single read can still lose a race: two concurrent first-saves both read `null`, both decide `'create'`, and one insert wins while the other hits the `UNIQUE(user_id, post_id)` violation. My Drizzle repo translates that Postgres `23505` into a typed `DuplicateSaveError`, and the service catches it and treats it as the no-op it logically is — the save exists, which is exactly what the caller wanted. So idempotency holds under concurrency, not just sequentially, without needing a transaction or advisory lock.

The alternative I deliberately didn't take is a single atomic upsert:

```sql
insert into saved_posts (user_id, post_id)
values ($1, $2)
on conflict (user_id, post_id)
do update set deleted_at = null, saved_at = now()
returning id, deleted_at;
```

This is race-free in one round trip and I considered it seriously. I didn't choose it because it moves the business rule (what counts as create vs. reactivate vs. no-op, and whether `saved_at` should bump on an already-active save) into SQL text, which means it's no longer testable without spinning up a database — directly against the spec's requirement that idempotency/reactivation/count behavior be verifiable in pure unit tests. My read-decide-write plus constraint-backstop approach costs an extra round trip on the write path in exchange for keeping the actual rule in TypeScript I can unit test in milliseconds.

**Hydrated flags (`hasSaved`, `savesCount`) via correlated subqueries in the page query itself.** Rather than fetching a page of posts and then issuing a second query (or N queries) to check save state, both the feed query and the saved-list query select two scalar subqueries per row: a `count(*)` over active saves for `savesCount`, and an `EXISTS` scoped to the viewer's own user id for `hasSaved`. Both are Postgres-correlated against the outer `posts.id`, so it's one round trip for a whole page, no N+1, and the counts are always computed fresh (never a cached counter that can drift). I looked at the alternative of a denormalized `saves_count` column on `posts` maintained by triggers or app-level increment/decrement, and rejected it — it reintroduces exactly the kind of state that can desync from the soft-delete rows, for a query that Postgres already answers correctly and cheaply with an index on `saved_posts(post_id) where deleted_at is null`.

**Check order is 401 → 404 → 403, everywhere.** Auth resolves first (no route is reachable unauthenticated). Then existence: a post that doesn't exist, or one a moderator soft-deleted, is a 404 regardless of whether the caller is enrolled in its course — "not found" doesn't leak whether a resource *would* have been forbidden. Only once a resource is confirmed to exist do I check whether the caller has rights to it, which is where 403 (`NOT_ENROLLED` / `NOT_MODERATOR`) comes in. This is also what the spec's own phrasing implies: 403 is described as "a post in a course they're not enrolled in" (i.e., an existing post), and 404 as "a post that doesn't exist" — so the ordering isn't arbitrary, it's what makes both status codes mean what the doc says they mean.

**Keyset (cursor) pagination, not offset.** Both the feed and the saved list paginate on `(timestamp, id) desc` with a base64url-encoded `{ts, id}` cursor, ties broken by `id` for a total order. I didn't use `OFFSET` because it degrades under concurrent inserts (rows shift under a moving offset, causing skipped or duplicated items across pages) and gets linearly slower on large offsets. Keyset pagination is stable under concurrent writes and every page is an indexed range scan. The cost is a slightly less "jump to page N" friendly API, which this product doesn't need — "load more" is the only UI affordance.

**The API returns machine error codes only (`NOT_ENROLLED`, `POST_NOT_FOUND`, `UNAUTHENTICATED`, `INVALID_CURSOR`, ...); the web app is the only layer that produces user-facing strings.** All domain errors are typed `ApiError` subclasses carrying their exact HTTP status, collapsed to `{ code }` bodies by one `onError` handler. The web `api-client.ts` throws a typed `ApiClientError` carrying that code, and the message catalog (`i18n/messages/{en,ar}.json`, under an `errors` key) is what turns `NOT_ENROLLED` into "You are not enrolled in this course" or its Arabic equivalent. This keeps the i18n boundary honest — no English leaks out of the API — and means adding a locale never touches server code.

## 2. Trade-offs and deliberate descopes

- **Likes, views, and comments are omitted entirely**, not half-implemented. They're mentioned as product color in the source doc but aren't in the numbered requirements list, and I'd rather ship a complete, well-tested Saved Posts slice than a partial version of three extra features.
- **No post detail page.** The feed and saved-list cards carry everything the requirements ask a card to show (title, body excerpt, author, date, save state, count); a detail route would be pure surface area with no new business logic to demonstrate.
- **`GET /api/users` is a demo-only endpoint** that exists purely to power the user switcher in the web UI (there's no real login, so something has to let a reviewer pretend to be Alice vs. Bilal vs. Mona). It's still behind the same auth resolve as every other route — it's a scoped concession to the "no real identity" constraint, not a hole in the auth model, and I called it out as demo-only rather than pretending it's a real endpoint.
- **No save-events audit table.** The single-row model I chose preserves *state* (is this currently saved, and when was it last saved/unsaved) but not a full *history* of every save/unsave event a user ever performed. That's a legitimate product feature (e.g., "saves over time" analytics) but it's a different data model — an append-only events table — layered on top of, not instead of, the current-state table. I scoped it out because the doc's requirement is "history preserved" in the sense of "re-save doesn't lose the row / doesn't duplicate," which the current schema already satisfies.
- **Zod response schemas aren't attached to routes on their error-status paths.** Elysia 1.4's native Zod (Standard Schema) response validation has an open upstream issue (elysia#1497) where attaching a `response` schema covering multiple status codes interacts badly with error responses. I validate all *inputs* (params/query/body) with Zod throughout, and error bodies are still shape-consistent (`{ code: string }`) by convention and by test coverage — I just didn't bolt a schema onto the framework's error path given the known bug, rather than fight an upstream issue on a take-home clock.
- **Save/un-save is read-decide-write (two round trips: read the existing row, then apply the write) rather than a single-statement upsert.** Covered in detail above — this is the direct cost of keeping the business logic testable in pure TypeScript instead of embedded in an `ON CONFLICT` clause. For this data volume and access pattern the extra round trip is not a meaningful cost; I'd revisit it if save/unsave became a hot path under real load.

## 3. What I'd do next with another day

- **Real session-based auth** (signed cookie or JWT with actual login) instead of the `x-user-id` header stub — the header is fine for a scoped take-home but isn't something I'd want near a real deployment.
- **Playwright e2e tests** covering the full save/un-save toggle and locale switch through the actual rendered UI, on top of the current unit/service/API/integration layers, to close the gap between "the API does the right thing" and "the button in the browser does the right thing."
- **A save-events audit table** alongside the current-state `saved_posts` table, to support "saves over time" analytics without disturbing the existing idempotency/reactivation model.
- **CI** running `bun run typecheck` and `bun run test` on every push — everything already runs from a clean checkout with no Docker required for tests, so wiring it into GitHub Actions is mechanical, I just didn't have the time slot for it in this box.
- **Rate limiting** on the save/un-save endpoints, since they're the only mutation surface a malicious or buggy client could hammer.
- **Post detail page and comments**, once likes/views/comments are actually in scope, to give Saved Posts something richer to point at than a card in a list.
